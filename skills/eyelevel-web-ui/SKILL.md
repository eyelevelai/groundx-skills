# EyeLevel Web UI

This skill produces front-end code for any web surface that should look and feel like EyeLevel — the GroundX dashboard, any other EyeLevel product UI, marketing pages, landing pages, demos, internal tools. The reference implementation is a TypeScript + React + MUI v5 app using Emotion (`sx` / `styled()`) with a custom theme, the Inter webfont (via Google Fonts), and a small set of branded wrappers around MUI primitives.

## What this skill produces

The output is always **standalone code** — imports resolve locally to the project being generated, never to a repo that doesn't exist on disk. A standalone marketing site, internal tool, demo app, or new product UI generated by this skill has its own `src/constants/`, its own `src/theme.ts`, its own `src/shared/components/`. Nothing reaches across into `groundx-ai-dashboard` at runtime.

At the same time, the output is **dashboard-compatible** — every project this skill produces is structured exactly like the GroundX dashboard repo: same constants barrel, same MUI theme shape, same `Gx*` / `Common*` wrapper layer around MUI primitives, same file-tree, same naming. A `<HeroSection>` written for a new marketing site should drop into the dashboard with no rewrites; a `GxCard`-using subtree lifted from the dashboard should drop into the new site the same way. The dashboard isn't a destination the skill imports from — it's the reference shape the skill replicates everywhere.

**This is one mode, not two.** Whether the user is extending the existing dashboard or starting a new project from scratch, the agent produces code with the same organizing principles. The only difference between "extending the dashboard" and "starting fresh" is whether the wrapper layer (`templates/`) is already present in the project; if it isn't, port it before writing the feature work. See *How to start a new project* below.

**Where creativity lives.** Inside that consistent skeleton, the agent has full latitude on what to build and how to compose it. New page types — a hero section, a pricing table, a testimonial wall, a partner logo grid, a careers page, a docs site shell, a sign-up flow — are normal feature work, not deviations from the skill. The brand stays consistent because the skeleton, the wrapper layer, and the visual invariants hold; the content and composition vary as widely as the project needs.

## First read the standards

**Before generating any visible code, read `../eyelevel-design-standards/SKILL.md`** (plus any of its references you need). That skill is the source of truth for:

- The palette (hex values, semantic roles, which pairings are allowed)
- Typography (Inter, weight scale, the ALL-CAPS-as-literal rule)
- Logos (which of the 4 files to use where)
- Brand principles (flat surfaces, green primary CTA + coral accent, navy not black)

This skill translates those invariants into MUI theme tokens, TypeScript constants, and React components. If you find a conflict between this skill and the standards skill, the standards skill wins — open a correction in this skill rather than diverging.

## Brand at a glance (web framing)

The standards skill has the full brand tour. The one-paragraph web version:

Green `#a1ec83` (`GREEN`) is the primary CTA fill — the hover-go-state on `CommonSubmitButton`, success chips, accent surfaces. Coral `#f3663f` (`CORAL`) is the secondary highlight accent — eyebrows, labels, the rest-state of the primary CTA. Body text is `#40496b` (`BODY_TEXT`); headings are navy `#29335c` (`NAVY`) — never true black. Section canvases use White, Gray `#f2f4f5`, Tint `#eff9fb`, Cyan `#c1e8ee`, Green, or Navy; cards on top are white with a `1px solid rgba(41, 51, 92, 0.1)` (`BORDER`) hairline and `20px` border-radius (`BORDER_RADIUS_CARD`). Buttons and pills use `200px` radius (`BORDER_RADIUS_PILL`) — the full-pill shape. No drop shadows, no elevation. Inter is applied at `<body>` via `MuiCssBaseline`; weight ladder is 400 / 600 / 700 / 800. The primary breakpoint is `md = 1100px` — bumped from MUI's default 900. Every EyeLevel project uses the same breakpoints (the structural-invariant story: a `<HeroSection>` written for one project drops into another only if both projects' breakpoint definitions agree).

## How to start a new project

The `templates/` directory contains the skeleton every EyeLevel-styled web project starts from. Its layout mirrors `src/` exactly — `templates/constants/` ports to `src/constants/`, `templates/shared/components/` ports to `src/shared/components/`, and so on — so the relative imports inside the template files (`../../constants`) resolve identically in both locations. **For any new project**, copy the templates into the project's `src/` tree (`cp -r templates/. src/` is enough) before writing feature code; that's what makes the project standalone (the imports resolve locally) and dashboard-compatible (the structure matches the dashboard's). For work inside an existing project that already has these files, skip this step.

| Template | Goes to | What it is |
| --- | --- | --- |
| `templates/constants/index.ts` | `src/constants/index.ts` | Barrel re-exporting everything from `constants.generated.ts` and `constants.chrome.ts`. After this lands, `import { NAVY, PADDING } from "@/constants"` resolves locally. |
| `templates/constants/constants.generated.ts` | `src/constants/constants.generated.ts` | Auto-generated brand tokens from `../../eyelevel-design-standards/tokens.json`. **Never hand-edit** — regenerate via `eyelevel-design-standards/scripts/generate-mirrors.mjs`. Identical across every EyeLevel project. |
| `templates/constants/constants.chrome.ts` | `src/constants/constants.chrome.ts` | Project-specific chrome tokens. The dashboard's version holds `drawerWidth`, `NAV_ICON_GREY`, `DISABLED_GREY`, etc. A marketing site's version may hold different values (or be empty). Each project owns its own. |
| `templates/theme.ts` | `src/theme.ts` | The MUI theme — Inter at body, `palette.primary.main = GREEN`, `disableElevation: true`, `palette.divider = BORDER`. |
| `templates/fonts.css` | `src/fonts.css` | `@import` of Inter from Google Fonts plus body-level `font-family` and OpenType feature settings (`ss01`, `cv11`, `cv01`). |
| `templates/ThemeProvider.tsx` | `src/ThemeProvider.tsx` | `GxThemeProvider` wiring theme + `CssBaseline` + fonts. |
| `templates/shared/components/*.tsx` | `src/shared/components/*.tsx` | Wrapper layer: `GxCard`, `GxPill`, `GxSectionHeader`, `GxButtonGroup`, `GxUsageCard`, plus `CommonSubmitButton`, `CommonCancelButton`, `CommonTextField`. Accessory layer: `CommonToolTip`, `CommonCloseIcon`, `DialogTitle`, `DropdownMenu`, `CopyToClipboard`, `DownloadButton`, `LoadingDots`, `UsageBar`, `VideoPlayer`, `CodeSnippet`. Port whichever the project uses. |

After porting, every component you write imports from the same paths the dashboard uses:

```tsx
import { CORAL, BORDER, BORDER_RADIUS_CARD, PADDING } from "@/constants";
import GxCard from "@/shared/components/GxCard";
import GxPill from "@/shared/components/GxPill";
import CommonSubmitButton from "@/shared/components/CommonSubmitButton";
```

The TypeScript path alias `@/` resolves to `src/` via `tsconfig.json` (`"baseUrl": "src"` or equivalent). For projects using a different bundler convention, use whatever local-path style the project's tsconfig already uses — what matters is that the import resolves to the **project's own** `src/`, not to a sibling repo. **Do not** import from `groundx-ai-dashboard` as a peer package.

**Mutual portability.** Because every project this skill produces shares the same skeleton, components are portable in either direction:

- A `<HeroSection>` built in a new marketing-site project drops into the dashboard's `src/shared/components/` (or wherever) and works — both repos resolve `@/constants` and `@/shared/components/GxCard` to the same shape.
- A subtree extracted from the dashboard (a settings page, a usage card, a modal) drops into a new project the same way.

This portability is the load-bearing reason for the structural conventions. If a new project chose a different folder convention (`components/` instead of `shared/components/`, or a different alias prefix), the portability would break — even if every visible pixel still matched the brand. Keep the conventions even when they feel small.

## The rules, distilled

These apply to every EyeLevel-styled web project — dashboard, marketing site, internal tool, demo, anything. They are what make code from this skill standalone and dashboard-compatible at the same time.

1. **No hex literals in component files.** Every color comes from a named token imported from `@/constants` (the project's own barrel re-exporting `constants.generated.ts`). New brand colors land in `eyelevel-design-standards/tokens.json` first, then propagate through `generate-mirrors.mjs` — never mint one locally.
2. **No `boxShadow` on surfaces, no MUI elevation, no gradient backgrounds.** Flat is the style. The single sanctioned gradient is the optional brand accent bar (see `eyelevel-design-standards/references/brand-principles.md` rule 10) and the upsell `gx-premium-button` variant.
3. **Inter at the body level.** Weight ladder is 400 / 600 / 700 / 800 — no other weights. Body copy is `BODY_TEXT`; headings are `NAVY` — never `#000`.
4. **No literal UPPERCASE hacks via `textTransform`.** Section labels are literal uppercase strings in JSX; buttons follow the per-component rule in `references/corrections.md > 6`.
5. **No hardcoded px breakpoints.** Use `theme.breakpoints.*` or sx responsive object syntax (`sx={{ p: { xs: 1, md: 5 } }}`).
6. **No MUI `<Card>`.** Use `GxCard` from `@/shared/components/`. If the project doesn't have `GxCard` yet, port it from `templates/shared/components/GxCard.tsx` — don't re-spell the styles inline.
7. **No raw `<Button>` for primary actions.** Use `CommonSubmitButton`. For segmented navigation, `GxButtonGroup`. For status labels, `GxPill` (not a button). Same porting rule applies — if the wrapper doesn't exist, port it.
8. **No hardcoded `fontSize: "16px"` on icons.** Use MUI size tokens: `small | medium | large`.
9. **Every interactive icon gets `aria-label`.** Decorative icons inside a labeled button get `aria-hidden="true"`.

The `Gx*` and `Common*` wrappers are not optional — they are how the brand is consistent across mediums. If you find yourself wanting to skip them ("this is just a marketing page, why do I need `GxCard` for the testimonial block?"), port the wrapper into the project's `src/shared/components/` and use it. The wrappers are tiny — `GxCard` is roughly 30 lines of JSX — and porting them takes less time than re-deriving their styles inline.

## Pre-return checklist

Before returning generated code:

**Project skeleton (only on the first commit to a new project).**

- [ ] `templates/constants/index.ts`, `constants.generated.ts`, `constants.chrome.ts` are at `src/constants/` and the `@/constants` alias resolves to them. `constants.chrome.ts` holds whatever project-specific chrome the project actually needs (a marketing site won't need `drawerWidth`; a dashboard does).
- [ ] `templates/theme.ts`, `fonts.css`, `ThemeProvider.tsx` are wired up at the project root.
- [ ] The `Gx*` and `Common*` wrappers used by the work are present at `src/shared/components/` (port any that aren't yet).

**Per-feature checks (every return).**

- [ ] Every color is imported from `@/constants`. No `#` characters in JSX or sx blocks.
- [ ] Every radius is a token (`BORDER_RADIUS_SM`, `BORDER_RADIUS`, `BORDER_RADIUS_2X`, `BORDER_RADIUS_CARD`, `BORDER_RADIUS_PILL`).
- [ ] Every breakpoint is `theme.breakpoints.*` or sx responsive syntax. No `@media (max-width: Xpx)` strings.
- [ ] Buttons use the right wrapper: `CommonSubmitButton` / `CommonCancelButton` / `GxButtonGroup` / `GxPill` — see `references/buttons.md` decision tree.
- [ ] Cards use `GxCard`, not MUI `<Card>` or inline `<Box sx={{ border, borderRadius, … }}>`.
- [ ] Text doesn't set `fontFamily` explicitly unless it's inside a `styled()` that resets inherited styles. Everything inherits Inter from `<body>`.
- [ ] Weight ladder is 400 / 600 / 700 / 800 — no other weights anywhere.
- [ ] Headings are `NAVY`; body is `BODY_TEXT`; no `#000` anywhere.
- [ ] ALL-CAPS labels are literal strings in JSX, not `textTransform`.
- [ ] Icons use `fontSize="small|medium|large"`, never a px value.
- [ ] Interactive icons have `aria-label`; decorative icons have `aria-hidden="true"`.
- [ ] No `boxShadow`, no `elevation` prop, no gradient backgrounds (unless it's the `gx-premium-button` variant or the optional brand accent bar).
- [ ] Logo lockup uses one of the two PNGs in `../eyelevel-design-standards/assets/logos/` — `eyelevel-logo-color.png` on light surfaces, `eyelevel-logo-white.png` on dark. "A VALANTOR COMPANY" is baked in; never re-typeset as a separate text element.
- [ ] Voice register matches `../eyelevel-design-standards/references/voice.md` — no "unleash", no neon-deck phrasing, no fabricated proof points.
- [ ] Photo backgrounds appear only when the user explicitly asked for imagery; otherwise the surface is solid color per the palette.
- [ ] If the work introduced a new component, it lives at `src/shared/components/` with a `Gx*` (primitive) or descriptive (composite) prefix and could be lifted into the dashboard repo without rewrites.
- [ ] If the work touches branding (logo placement, color choices, new type styles), it conforms to the standards skill — no local overrides.

## When in doubt

When you can't decide between two approaches, the one that results in **fewer tokens crossed into the component** and **more configuration up in the theme** is almost always correct. EyeLevel's visual system is calm and opinionated — aim for component files that read like plain structural JSX, with styling coming from theme overrides, sx tokens, and the Gx* wrappers.

If a pattern truly doesn't exist yet, add it to the theme or to `templates/shared/components/` (in your response) and reference that addition — don't open-code it in a one-off sx block. If the missing pattern is a brand-level decision (a new color, a new weight), add it to the standards skill first.
